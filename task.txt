## Bug 1 — User messages never saved to DB

**File:** `index.js` and `lib/agent.js`

**Problem:** The old `messageCreate` handler called `saveMessage` for the user's message before calling the agent. The new handler removed this. `agent.js` only saves assistant and system replies — never the user's message. So `issue_messages` is missing every user message. Future turns have empty history. Staff DB queries have gaps.

**Fix in `index.js`:** In the `messageCreate` handler, add `saveMessage` call for the user's message BEFORE calling `runAgent`. Import `saveMessage` from `lib/issues.js` at the top of `index.js` if not already imported (it should already be there).

The `messageCreate` handler should look like this:

```js
client.on('messageCreate', async message => {
  if (message.author.bot) return;
  if (!message.channel.isThread()) return;

  const thread = message.channel;
  if (thread.parentId !== process.env.BAD_REPORT_CHANNEL_ID) return;

  const content = message.content.trim();
  if (!content) return;

  const { data: issue } = await supabase
    .from('issues')
    .select('*')
    .eq('thread_id', thread.id)
    .maybeSingle();

  if (!issue) return;
  if (issue.status === 'resolved' || issue.status === 'closed') return;

  // Save user message BEFORE running agent so history is complete
  await saveMessage({
    issueId:      issue.id,
    role:         'user',
    content,
    discordMsgId: message.id
  });

  await thread.sendTyping();
  await runAgent(client, thread, issue, content);
});
```

**Fix in `lib/agent.js`:** Remove the `saveMessage` import and all calls to `saveMessage` inside `runAgent` for the user's message. The agent should only save assistant and system messages (replies and escalation logs). User messages are now saved in `index.js`.

Keep these saves in `agent.js`:
- CASUAL reply save (assistant role) ✅
- STATUS reply save (assistant role) ✅  
- Final answer save (assistant role) ✅
- Escalation log save (system role) ✅

Remove any save that saves the user's incoming message inside `agent.js`.

## Bug 2 — Rewriter hallucinates reasons and responder answers with no context

**File:** `lib/rewriter.js` and `lib/responder.js`

**Problem:** The rewriter can say "Already answered in conversation" when there is no conversation history at all. It hallucinates a reason. Then the responder, seeing `needsRagWasAttempted: false`, allows the LLM to answer from "conversation history" — but the LLM uses its training data instead. This produces confident wrong answers.

**Root cause:** The rewriter's prompt doesn't check if history is actually empty before claiming something was "already answered". The responder trusts the rewriter's `needsRag: false` decision without validating whether history actually contains relevant content.

**Fix in `lib/rewriter.js`:** Add a pre-check before calling the LLM. If history is empty, `needsRag` must be `true` for QUESTION and COMPLAINT intents — there is nothing to answer from. Update the function signature and add this guard:

```js
async function rewriteQuery(userMessage, history, intent = 'QUESTION') {
  // If no conversation history exists, we MUST search RAG for questions/complaints
  // Cannot claim "already answered" with empty history
  if (history.length === 0 && ['QUESTION', 'COMPLAINT', 'UNCLEAR'].includes(intent)) {
    // Still rewrite the query, but force needsRag: true
    // Use a simplified prompt that only does query rewriting, not the needsRag decision
    const simpleResult = await rewriteQueryOnly(userMessage);
    return { ...simpleResult, needsRag: true, reason: 'No history — must search docs' };
  }

  // ... rest of existing rewriteQuery logic unchanged
}
```

Add this helper function above `rewriteQuery`:

```js
const REWRITE_ONLY_PROMPT = `You are a search query optimizer.
Convert the user message into a clean 3-8 word semantic search query.
Remove emotional language, filler words, pronouns.
Keep technical terms, product names, action verbs.
Respond with ONLY the query string. No explanation, no JSON, no quotes.

Examples:
"what payment methods do you accept?" → accepted payment methods
"my app keeps crashing on login" → app crash login authentication
"how do I cancel my subscription?" → cancel subscription steps
"ugh nothing is working" → product functionality issues`;

async function rewriteQueryOnly(userMessage) {
  try {
    const result = await chat(REWRITE_ONLY_PROMPT, [
      { role: 'user', content: userMessage }
    ]);
    const query = result.trim().replace(/^["']|["']$/g, '').slice(0, 100);
    return { query, needsRag: true, reason: 'No history — must search docs' };
  } catch (err) {
    console.error('[rewriter] rewriteQueryOnly failed:', err.message);
    return { query: userMessage.slice(0, 100), needsRag: true, reason: 'Fallback' };
  }
}
```

Update the call in `lib/agent.js` to pass `intent`:
```js
const { query, needsRag, reason } = await rewriteQuery(userMessage, context.history, intent);
```

**Fix in `lib/responder.js`:** When `needsRagWasAttempted` is `false` but history is also empty or short (less than 2 messages), escalate instead of attempting to answer from history. Add this check:


## Bug 3 — UNCLEAR intent falls through to full pipeline unnecessarily

**File:** `lib/agent.js`

**Problem:** UNCLEAR messages (too vague to classify) go through the full Layer 2 → Layer 3 → Layer 4 pipeline. This burns LLM calls and usually ends in escalation anyway. Better to ask for clarification immediately.

**Fix:** Add UNCLEAR handling after STATUS in `runAgent`:

## Bug 5 — JSON parsing fragility in `rewriter.js`

**File:** `lib/rewriter.js`

**Problem:** `JSON.parse` on LLM output is fragile. The LLM can return:
- Unescaped quotes inside strings: `{"query": "user's broken app"}` → parse error
- Trailing commas: `{"query": "foo", "needsRag": true,}` → parse error  
- Extra explanation text before/after JSON → parse error
- The string `"null"` instead of `null` for the query field

The current regex only strips markdown code fences. Not enough.

**Fix:** Replace the JSON parsing section in `rewriteQuery` with a robust parser:

```js
// Robust JSON extraction — handles LLM quirks
function extractJSON(text) {
  // Find the first { and last } to extract just the JSON object
  const start = text.indexOf('{');
  const end   = text.lastIndexOf('}');
  if (start === -1 || end === -1) return null;

  let jsonStr = text.slice(start, end + 1);

  // Fix common LLM JSON mistakes
  // Remove trailing commas before } or ]
  jsonStr = jsonStr.replace(/,\s*([}\]])/g, '$1');

  try {
    return JSON.parse(jsonStr);
  } catch {
    return null;
  }
}
```

Replace the current parsing block in `rewriteQuery`:

```js
// Replace this:
const cleaned = result.replace(/```json\n?/g, '').replace(/```\n?/g, '').trim();
const parsed  = JSON.parse(cleaned);

// With this:
const parsed = extractJSON(result);
if (!parsed) {
  console.warn('[rewriter] Could not parse LLM response as JSON, using fallback');
  return {
    query:    userMessage.slice(0, 100),
    needsRag: true,
    reason:   'JSON parse failed — using raw message as query'
  };
}
```

Also fix the `"null"` string issue in the validation block:

